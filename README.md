# ניהול צי סירות (Charter Yacht Fleet Management)

אפליקציית Next.js לניהול צי סירות שכירות, עם התחברות אמיתית ומאובטחת דרך Supabase Auth ו-Row Level Security (RLS) שאוכף את ההרשאות ישירות במסד הנתונים.

## תפקידים והרשאות

| תפקיד | גישה |
|---|---|
| **ניהול (management)** | רואה ועורך את כל הסירות, המשתמשים והנתונים בצי |
| **קפטן (captain)** | מחובר לסירה אחת, רואה ועורך את כל הנתונים של הסירה שלו בלבד |
| **בעלים (owner)** | מחובר לסירה אחת, רואה רק רשומות **שאושרו** על ידי הניהול בכל מודול שתומך באישור (כרגע: הוצאות, תקלות טכניות, הזמנות, צוות) |

ההרשאות אינן רק ברמת ה-UI — הן אוכפות ב-Postgres דרך Row Level Security, כך שגם קריאה ישירה ל-API של Supabase לא תחשוף נתונים שלא מותרים למשתמש.

### זרימת אישור (Approval workflow)

מודולי ההוצאות, התקלות הטכניות, ההזמנות, הצוות והמסמכים (כולל ציוד בטיחות, שהוא בעצם מסמך מסוג `safety`) מיישמים זרימת אישור אמיתית: רשומה שנוצרת על ידי קפטן מקבלת סטטוס `pending`, רשומה שנוצרת על ידי ניהול מאושרת אוטומטית (`approved`). ניהול יכול לאשר רשומה ממתינה (כפתור "אשר"). בעלים רואה רק רשומות מאושרות — האכיפה היא ב-RLS ולא רק ב-UI, כולל על תמונות/קבצים מצורפים ב-Storage (קבלות, תמונות תקלה, דרכוני אורחים, קבצי צוות). Trigger משותף בשם `prevent_self_approval` חוסם קפטן מלאשר את עצמו גם דרך קריאת API ישירה (מחובר לכל טבלה שתומכת באישור).

לתקלות טכניות יש בנוסף סטטוס תפעולי נפרד (`op_status`: לא התחיל/ממתין/בטיפול/הושלם/בוטל) שניתן לקדם בלחיצה — זה עוקב אחרי התקדמות הטיפול, בנפרד מסטטוס האישור.

**משכורות צוות הן מקרה מיוחד**: קפטן יכול ליצור ולערוך אנשי צוות, אבל לא לראות את שכרם (רק ניהול ובעלים רואים סכומי משכורת). מדובר בהגבלה ברמת עמודה בודדת, ש-RLS רגיל (row-level) לא יכול לאכוף ישירות — הפתרון: הרשאת ה-SELECT על טבלת `staff` הבסיסית מוגבלת לניהול בלבד, וקריאה מתבצעת דרך view בשם `staff_visible` שמשחזר את לוגיקת הראייה הרגילה (ניהול/קפטן-בסירה-שלו/בעלים-מאושר-בלבד) אך מאפס את שדה השכר ללא ניהול/בעלים. ה-view הזה **אינו** security-invoker בכוונה, כדי שיוכל להעניק לקפטן גישה לשורות שה-RLS של הטבלה הבסיסית חוסם ממנו ישירות.

**מסך אישורים מרוכז** (`/approvals`, ניהול בלבד): כל הרשומות הממתינות מכל הסירות במקום אחד (כולל מסמכים), עם אפשרות סינון לפי סירה ואישור/דחייה ישירות. תג עם מספר הפריטים הממתינים מופיע ליד הקישור בתפריט העליון.

## מודולים

לכל סירה: מסך "סקירה" עם סטטיסטיקות (הוצאות שנתיות מול תקציב, תקלות פתוחות, אנשי צוות), מפרט מלא (דגם/אורך/רוחב/שוקע/דגל/מקום עגינה/מספר רישוי/MMSI עם קישור למיקום חי ב-MarineTraffic), לוגו ותמונת סירה, מסמכים (ביטוח/רישיון/רישום, זרימת אישור) עם העלאת קבצים ל-Supabase Storage, תקלות טכניות + ציוד בטיחות (סיווג, אזור, ספק, הצעת מחיר, סטטוס תפעולי, זרימת אישור), הזמנות/יומן זמינות (לוח שנה ויזואלי, סוג שימוש, דרכוני אורחים, הורדת רשימת צוות ואורחים להדפסה, זרימת אישור), וצוות ושכר (פרטי איש צוות, קורות חיים, משכורת מוסתרת מקפטן, זרימת אישור).

**כספים** (כמה תתי-לשוניות): הוצאות עם קבלות מצולמות וזרימת אישור; מצב חשבון בנק + הכנסות בפועל; תנועות מזומן (משיכה מפחיתה אוטומטית את יתרת הבנק); חשבוניות (תצוגת הוצאות מאושרות עם קבלה, לפי חודש, עם אפשרות הדפסה); הכנסות עתידיות; דוח תקופתי (הכנסות/הוצאות/תזרים נטו/פילוח לפי קטגוריה לטווח תאריכים, כולל גרף עוגה); ותקציב שנתי לפי קטגוריה עם תתי-קטגוריות.

**קטלוג מכירה**: לסירה מסוג "יד שנייה למכירה" (`boat_type = for_sale`, נבחר בטופס הסירה) יש לשונית "קטלוג" עם מחיר מבוקש וגלריית תמונות, במקום לשוניות התפעול היומיומי (תחזוקה/הזמנות/צוות מוסתרות לסוג הזה).

**הכנות להפלגה**: רשימות קניות (בניית סל עם כמות/יחידה/תמונה, שליחה לצוות, סימון פריטים כנאספו, קישור לטיול) והזמנות הסעה (מספר טיסה/זמן נחיתה/סוג רכב/איסוף/הורדה, סימון כתואם). **שימו לב לחריגה מהכלל הרגיל**: כאן דווקא ה**בעלים** הוא זה שיוצר בקשות (מזמין קניות/הסעה) יחד עם הניהול, בעוד שקפטן רק מסמן פריטים כנאספו ולא יוצר/מוחק רשימות — הפוך מכל שאר המודולים שבהם הבעלים read-only. זה תואם בדיוק את ההתנהגות באפליקציית הדמו המקורית.

**דוחות חודשיים**: ניהול מנפיק "תמונת מצב" קפואה (financial/technical) לחודש נבחר — הנתונים נשמרים כ-JSON בזמן ההנפקה, כך שדוח שהונפק לא משתנה גם אם ההוצאות/תקלות שבבסיסו נערכות אחר כך. קפטן ובעלים יכולים לקרוא דוחות (ללא יכולת עריכה/מחיקה); אין להם זרימת אישור — ניהול הוא המחבר היחיד מטבע הדבר.

**הושמט בכוונה**:
- התאמת בנק אוטומטית מקובץ CSV (Bank reconciliation) — פיצ'ר מורכב יחסית (פענוח CSV + התאמה מטושטשת), נדחה למודול נפרד.

## סריקה חכמה (AI)

שני מקומות באפליקציה משתמשים ב-Claude (Anthropic) כדי לסרוק תמונות/מסמכים ולמלא שדות אוטומטית:
- **הוספת הוצאה** (בכספים ובסקירת הסירה): צילום/העלאת קבלה ממלא תיאור, סכום, תאריך, מספר חשבונית וקטגוריה.
- **הוספת חוזה MYBA** (ביומן, לא זמין לסירות פרטיות): העלאת חוזה חתום סרוק (תמונה או PDF) ממלאת שם שוכר, תאריכים, אזור הפלגה, שכר השכרה, מקדמה, תאריך תשלום ומספר הזמנה — ויוצרת אוטומטית טיול ביומן, רשומות הכנסה עתידית (שכר + מקדמה), ואת קובץ החוזה במסמכים (מסוג `myba_contract`, בתיקייה `myba_contracts/<שנה>`), כשכולם מקושרים זה לזה דרך `booking_id`.

זה דורש משתנה סביבה `ANTHROPIC_API_KEY` (מפתח API של Anthropic, בעל יתרה) שיש להוסיף ב-Vercel (Project Settings → Environment Variables). בלי המפתח, הסריקה פשוט לא זמינה והשדות ממולאים ידנית כרגיל.

- Storage bucket פרטי בשם `documents` עם מדיניות גישה תואמת

הקובץ השני (מודול הוצאות ותקציב) מוסיף:

- Storage bucket פרטי בשם `receipts` לתמונות קבלות

הקובץ השלישי (מודול תקלות טכניות + ציוד בטיחות) מוסיף:
- טבלת `issues` (מחליפה את `maintenance_records` הישן)
- הוספת קטגוריה `safety` לטבלת `documents` הקיימת + עמודת `last_checked_date`, לשימוש ציוד בטיחות
- Storage bucket פרטי בשם `issue-attachments` לתמונות תקלה והצעות מחיר

הקובץ הרביעי (שדרוג מודול הזמנות) מוסיף:
- מחליף את סטטוס ההזמנה הישן (`booking_status`) בזרימת האישור (`approval_status`) המשותפת
- עמודות סוג שימוש, מספר אורחים ואזור הפלגה
- טבלת `booking_guests` (דרכוני אורחים לכל הזמנה) + Storage bucket פרטי בשם `booking-guests`

הקובץ החמישי (מודול צוות ושכר) מוסיף:
- טבלת `staff` (SELECT ישיר מוגבל לניהול בלבד) + ה-view `staff_visible` שמסתיר שכר מקפטן
- Storage bucket פרטי בשם `staff-files` לתמונות פרופיל וקורות חיים

הקובץ השישי (שאר תתי-הלשוניות של כספים) מוסיף:
- טבלאות: `incomes` (בפועל/עתידי), `cash_transactions` (משיכה/שימוש), `bank_balances` (שורה אחת לכל סירה)
- פונקציית `apply_cash_withdrawal` (SECURITY DEFINER): מאפשרת לקפטן לרשום משיכת מזומן שמפחיתה אוטומטית מיתרת הבנק, גם שההרשאה הישירה על `bank_balances` מוגבלת לניהול בלבד

הקובץ השלושה-עשר מוסיף:
- מוחק את טבלת `recurring_expenses` (הפיצ'ר "הוצאות קבועות" הוסר)

הקובץ השביעי (מודול מסמכים - זרימת אישור) מוסיף:
- עמודות `status`/`approved_by`/`approved_at` לטבלת `documents` הקיימת + ה-trigger שמונע אישור עצמי
- מדיניות RLS מעודכנת (כולל על ה-Storage bucket `documents`) כך שבעלים רואה רק מסמכים מאושרים, כולל את הקובץ עצמו

הקובץ השמיני (מודול קטלוג מכירה) מוסיף:
- Enum `boat_type` (commercial/private/for_sale) + עמודות `boat_type`/`sale_price` לטבלת `boats`
- טבלת `catalog_photos` + Storage bucket פרטי בשם `catalog`

הקובץ התשיעי (מודול הכנות להפלגה) מוסיף:
- טבלאות: `shopping_lists`, `shopping_list_items`, `transfer_requests`
- מדיניות RLS ייחודית שבה לבעלים יש הרשאת כתיבה (יצירה/מחיקה), בניגוד לכלל הרגיל
- Storage bucket פרטי בשם `shopping` לתמונות פריטי קניות

הקובץ העשירי (מודול דוחות חודשיים) מוסיף:
- טבלת `reports` עם עמודת `snapshot jsonb` (תמונת מצב קפואה בזמן ההנפקה)
- RLS: הנפקה/מחיקה לניהול בלבד, קריאה לכל התפקידים בסירה

הקובץ האחד-עשר (מפרט סירה מורחב + תמונות) מוסיף:
- עמודות לטבלת `boats`: `beam_meters`, `draft_meters`, `flag`, `berth`, `mmsi`, `logo_path`, `image_path`
- Storage bucket פרטי בשם `boat-photos` (לוגו + תמונת סירה), מדיניות זהה לדפוס של קבצי סירה אחרים

הקובץ השנים-עשר מוסיף:
- עמודת `parent_boat_id` לטבלת `boats` — מאפשר לסמן סירה כתת-סירה של סירה אחרת (למשל טנדר/סירת אחות), מוצג עם הזחה ברשימת הצי

הקובץ הארבעה-עשר מוסיף:
- ערך `received` ל-enum `cash_tx_type` — קטגוריית "קבלת מזומן ביד", בנוסף למשיכה ולשימוש

הקובץ החמישה-עשר מוסיף:
- עמודת `notes` לטבלת `expenses`

הקובץ השישה-עשר מוסיף:
- ערך `myba_contract` ל-enum `document_type`
- עמודת `booking_reference` לטבלת `bookings`
- עמודת `booking_id` לטבלאות `incomes` ו-`documents` — מקשרת רשומת הכנסה עתידית / קובץ חוזה לטיול הספציפי שנוצר ממנו

### 3. יצירת משתמש הניהול הראשון

אין הרשמה עצמאית באפליקציה — משתמשים נוצרים רק דרך מסך "משתמשים" של תפקיד הניהול. כדי ליצור את משתמש הניהול הראשון (chicken-and-egg):

1. ב-Supabase Dashboard → **Authentication → Users → Add user**, צרו משתמש עם אימייל וסיסמה.
2. ב-**SQL Editor** הריצו (מחליפים את האימייל):

```sql
update public.profiles set role = 'management', boat_id = null
where id = (select id from auth.users where email = 'you@example.com');
```

מרגע זה, המשתמש הזה יכול להתחבר לאפליקציה ולנהל את שאר הצוות דרך מסך "משתמשים".

### 4. הרצה מקומית

```bash
npm install
npm run dev
```

האפליקציה תעלה על [http://localhost:3000](http://localhost:3000) ותפנה אוטומטית למסך ההתחברות.

## מבנה טכני

- **Next.js App Router** + Server Actions לכל פעולות הכתיבה (יצירה/עדכון/מחיקה) — אין API routes נפרדים מלבד הורדת מסמכים.
- **`@supabase/ssr`** לניהול session מבוסס-cookies, כולל middleware (`src/middleware.ts`) שמרענן את ה-session ומגן על נתיבים לא מחוברים.
- **RLS כקו ההגנה האמיתי**: קליינט השרת הרגיל (`src/lib/supabase/server.ts`) פועל תמיד בהקשר המשתמש המחובר, כך שגם אם קוד ה-UI מפספס בדיקת הרשאה, מסד הנתונים לא יחזיר או ישנה נתונים אסורים. קליינט ה-service role (`src/lib/supabase/admin.ts`) משמש אך ורק ליצירה/מחיקה של משתמשים ומיובא בצד שרת בלבד.
- כל טבלה משתמשת ב-`type` (לא `interface`) עבור טיפוסי ה-Row/Insert/Update — נדרש כדי שההיסק הגנרי המורכב של `@supabase/supabase-js` לא יקרוס ל-`never`.
